## 0. 协作约束（最重要，先读这一条）

我（用户）是一个**不懂技术**的人，**脾气比较急**，提需求时**描述未必清晰、也不会用专业术语**。

和我协作时，请始终遵守以下原则：

1. **先理解，再动手。** 把我说的"大白话"先翻译成产品逻辑，确认我真正想要的是什么，而不是逐字照做。遇到模糊的地方，用"产品经理"的思维去补全，而不是直接问我一堆技术问题。
2. **用人话沟通。** 不要堆砌技术名词（OCR、API、状态管理这种词能不说就不说）。要说也要先用生活化的比喻解释一句。
3. **场景化表达。** 用"老人/子女会怎么用"的具体场景来描述方案，而不是抽象的功能清单。
4. **方案要可执行。** 直接给出"我建议这么做"的最终方案，并简要说明"为什么"，而不是把选择题甩回给我。
5. **照顾我的情绪。** 我可能会因为没看懂或者结果不对而着急。沟通要有耐心、给确定性，少让我做决定、多替我兜底。
6. **重大改动先打招呼。** 涉及删数据、改方向、推翻已有功能这类大动作，先用一句话告诉我影响，再动手。
7. **改动前先给我过目，等我说"开始做"再动手。** 每次收到需求，先把"我打算怎么改"用清单列出来给我确认，**不要直接写代码**。除非我在当次对话里明确说了"开始做"或类似的指令。
8. **不要截图验证效果。** 改完代码不需要启动浏览器截图给我看，我会自己刷新预览。截图会浪费 token，除非我主动要求。

> 一句话总结：**把我当成产品负责人，你是既懂产品又懂技术的执行者。我负责拍板方向，你负责把模糊需求变成靠谱的东西。**

## 1. 产品概述

「长青伙伴」是一款面向**老龄用户**的**用药提醒 + 健康陪伴** App。

**核心目标：** 用"种树 / 养宠物"的游戏化养成模式，降低老人对吃药的抵触情绪，把"被动吃药"变成"主动照顾我的小树"。

**核心痛点：**
- 老人经常忘记吃药。
- 市面上的 App 太复杂，不适老。
- 老人对"吃药"这件事本身有心理抵触，不喜欢被当成"病人"。

---

## 2. 角色分工（产品的灵魂设计）

> ⚠️ **当前决定（2026-06）：先做"单端合并"版，暂不拆分老人端 / 子女端。** 一套界面、一份数据，开发量减半。等以后真有"子女远程管理"的需求再拆分。

- **当前形态：** 一个 App 同时承载"养成 + 确认吃药"和"拍照录药 + 管理"，但**首页依然以养成为主**，把管理类入口降权、收纳到次级页面（如"用药""设置"标签页），保持老人进门没压力。
- **预留方向（暂不做）：** 未来可拆出独立的"子女端（管家版）"，负责远程录药、查看数据。

> 设计原则：**把"治疗"的压迫感藏起来，把"生活陪伴"的感觉凸显出来。**老人不应该一进门就被"添加新药"提醒"你是个病人"。

---

## 3. 功能模块

### 3.1 首页：养成陪伴空间（老人端核心）
- **视觉中心是一棵"长青树"（或宠物）**，不是药单。树的状态（精神 / 有点渴 / 升级）反映服药进度。
- 顶部一句温馨问候引导，如："小树今天需要浇水（吃药）啦，它会陪你一起变健康哦。"
- 一个醒目的大按钮 = "确认已服药"（= 给小树浇水）。
- **例外：** 如果还没有任何服药计划，首页才温和地引导去添加。
- 连续坚持 → 树升级（类似多邻国连续打卡）；断药 → 树会"渴"，用软性提醒引导，不指责。

### 3.2 药物与诊单管理（拍照录入）
- **入口主要在子女端**，老人端默认隐藏。
- **一键拍照录入：** 拍药单 → 自动识别药名、剂量、频率、开始日期 → 展示精简确认卡 → 一键确认 / 发给子女修改。
- **智能排期：** 用"从今天开始 / 从明天开始"两个大按钮，避免复杂日历操作。
- **防呆：** 老人拍照容易抖、模糊，拍到清晰一帧就自动锁定处理，不要求对得很准。

### 3.3 反馈与"哄哄"机制
- 每次确认吃药 → 出现夸赞动画（如"奶奶真棒，小树长高了！"）+ 阳光值奖励。
- 超时未吃 → 推送"软话"提醒，用儿女关心的口吻，不指责（如"我知道这药有点苦，但吃完身体棒棒，儿女也放心呀"）。

### 3.4 健康档案（归档）
- 体检报告、化验单、诊单 → 拍照存入"电子病历本"。
- **只做影像归档，不做复杂数值解析**（容易错），方便复诊时调出给医生看，按时间轴排列。

## 4. 视觉与交互规范
- **主色调：** 充满生机的"活力绿"。Icon 也是绿色、充满生机。
- **字体 / 按钮：** 全局超大字号，超大点击面积，尽量减少二级菜单。
- **交互原则：** 尽量"单点触控"，能不打字就不打字。
- **老人模式：** 可开关的极简界面，只保留养成和吃药。

## 5. 技术方案（已敲定）

> 背景：用户是前端，不想维护服务器；目标用户是**用 iPhone 的老人**，产品面向**国内**。曾用过 Prisma + Neon，但 Neon 在国外、需翻墙，不适合。曾试过 AI 生成原生 iOS App，体验差（页面乱跳）。

### 5.1 端形态：微信小程序 ✅
选小程序而非原生 App / 网页的核心理由：
1. **老人零门槛**：微信里直接打开 / 扫码，不用 App Store、不用 Apple ID、不用密码。
2. **分享方便**：家人转发一下老人就能用。
3. **AI 生成稳定**：小程序页面路由规整、能即时预览，避开原生 App"页面乱跳"的坑。
4. 成本低、迭代快、无上架审核等待。

### 5.2 后端 / 数据：微信云开发（CloudBase）✅
- 微信官方自带的"免费后端"，**无需自建服务器、无需翻墙、机房在国内**（快且稳）。
- 提供三样关键能力：**数据库**（药物、服药记录、小树状态）、**文件存储**（药单/化验单照片）、**自动登录**（打开即知"是谁"，老人免注册免密码）。
- 免费额度足够起步。
- ❌ 不用 Neon / Supabase 等海外数据库（翻墙问题）；不自建服务器 + Prisma（运维太重）。

### 5.3 提醒可靠性方案（核心命根子）
小程序自身推送偏弱，采用**双保险**：
- **主力：一键同步到 iPhone 自带"日历 / 提醒事项"** → 系统级闹钟，到点必响、关机重开仍在，最可靠。
- **辅助：微信订阅消息**（医疗健康类目可申请长期订阅，需企业资质，**先不折腾，靠日历即可**）。

### 5.4 数据结构（云开发文档库，无需复杂表关系）
- **用户**：昵称、小树等级、连续天数、总服药次数。
- **药物**：药名、剂量、频率、时间点、饭前/饭后、开始日期、长期/短期（含结束日）。
- **服药记录**：日期、药物、服用时间、是否已服。
- **照片档案**（P2）：照片、日期、类型。

### 5.5 落地节奏
1. **第一步**：先用本地数据把**界面 + 点击流程**跑顺（种树 / 提醒 / 夸夸），快速看到能点能跑的版本。
2. **第二步**：接入云开发（换手机不丢数据）+ 同步日历提醒 + 拍照识别。
3. **第三步**：健康档案归档等 P1/P2 功能。

## 7. 适老化优化清单（走查结论，2026-06）

> 以老人视角逐页走查得出，按优先级排列。

### P0（最影响老人能不能用）
- [ ] **配置改"点选"，消灭手填**：时间段用大卡片点选（早餐后/午餐后/晚餐后/睡前），剂量用 `－ 数字 ＋` 步进器，单位用大按钮选（片/粒/支）。吃药本质是定时任务，"输入文字频率"不够直观。
- [ ] **已服打卡防误触**：点已打卡的药不能直接撤销，需二次确认弹窗，避免老人手抖误操作。
- [ ] **状态字放大加深**："待服/已服"改成"还没吃/已吃了"，字号更大、颜色更深，老人一眼看清。
- [ ] **夸夸常驻直到主动关闭**：原来 2.2 秒自动消失，老人反应慢没看清就没了，改为"轻触关闭"。
- [ ] **今日用药按早/中/晚分组**：按时间段分组显示并高亮当前时段，老人知道"现在该吃哪个"。

### P1（更顺手、更懂老人）
- [ ] 编辑/删除按钮放大（现在 emoji 图标太小，老人容易点错）。
- [ ] 底部导航加图标（老人靠图形认路，不靠文字）。
- [ ] 老人模式时底部标签减为 2 个（养成 + 记录）。
- [ ] 全局文案口语化（"待服→还没吃"、"已服→已吃了"、"退出登录→换个人用"）。
- [ ] "同步至日历提醒"改为"让手机到点提醒我吃药"。

### P2（锦上添花）
- [ ] 重要操作加震动反馈（浇水确认时）。
- [ ] 树等级显示形态名（"种子"）而非数字（"Lv.1"）。
- [ ] 拍照确认卡：超大字 + 对/不对两个大按钮 + "让家人帮我改"。
